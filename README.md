# AI Storefront Benchmark

**AI Software Engineers Compared: Building the Same Storefront with Three Models**

---

## Overview

This benchmark explores how close modern AI models are to functioning as autonomous software engineers by tasking three different models with building the exact same e-commerce application from a shared set of requirements — independently, without access to each other's work.

The competition is organized by **[Cindy Singleton](https://github.com/cindy-pi)** as part of an industry research paper examining the real-world software engineering capabilities of modern large language models. Each model is orchestrated through **[Digiswarm AI](https://digiswarm.ai)**, using its liaison AI agent — code-named **Paul** — who receives the project brief, decomposes it into tasks, and directs that model's coding agents to deliver a complete product.

Digiswarm AI was chosen specifically because it allows the underlying model to be swapped without changing the agent framework, tooling, context, or skills. This means the only variable between runs is the model itself — making the comparison as controlled as possible.

All three runs use the **Digiswarm Starter Plan** (1 agent at a time). Each session has 4 agents configured, but they execute sequentially — build times in this benchmark reflect single-agent sequential execution. Higher Digiswarm plans support up to 5 parallel agents, which would reduce wall-clock time considerably; that variable will be explored in a follow-up paper.

The result is a transparent, reproducible experiment comparing commercial cloud-hosted models against a powerful self-hosted open model — evaluated on timing, cost, and real-world site quality.

## How It Works

### 1. The Liaison Agent (Paul)

Each competing model is given its own instance of **Paul** — the [Digiswarm AI](https://digiswarm.ai) liaison agent responsible for receiving the project brief, decomposing it into tasks, and directing that model's coding agents to produce a working product. Paul is the first point of contact and the orchestrator; the coding agents do the actual implementation.

The same prompt is given to each Paul instance. No Paul has visibility into what another Paul's team produces.

### 2. The Competition Prompt

Each Paul receives an identical brief. The full prompt is documented in [PROMPT.md](PROMPT.md).

The project is called **Fizban's Wands** — a whimsical magical wand shop e-commerce site inspired by the Dragonlance Chronicles, deployable as a static React app on GitHub Pages.

### 3. Independent Execution

Each model's agents work from scratch. No shared code, no cross-pollination, no hints about competitor approaches. The only shared artifact is the prompt.

### 4. Deployment Standard

All three submissions must deploy automatically to GitHub Pages via GitHub Actions on every push to `main`. This ensures each site is always live and accessible for review. The required repo configuration is documented in [REPO_SETUP.md](REPO_SETUP.md) and is part of the competition brief.

### 5. Scoring

Once all three teams deliver their sites, scoring happens in two dimensions:

**Operational Metrics** (objective)
- Total wall-clock time from prompt to working build
- Total token cost (API spend or compute equivalent for local)

**Site Quality Score** (blind peer review)
- Each of the three models independently reviews **all three sites** without knowing which model built which
- Each reviewer scores each site across three dimensions:

| Dimension | Description |
|---|---|
| **Originality** | Is the design fresh and creative, or generic and predictable? |
| **Quality** | Is the implementation solid, complete, and functionally correct? |
| **Elegance** | Is the UI polished, cohesive, and pleasant to use? |

- Each dimension is scored 1–100
- The three dimension scores are averaged into a single **site score** per reviewer
- The three reviewer scores (one per model) are averaged into the **final score** for that site
- The model with the highest final score **wins**

Full scoring methodology and rubric are in [SCORING.md](SCORING.md).

## Why This Matters

This benchmark is the basis of an industry research paper by [Cindy Singleton](https://github.com/cindy-pi) examining AI-driven software development. It is designed to provide reproducible, comparable data for developers, AI researchers, startups, and engineering teams interested in:

- The real-world software engineering capabilities of large language models
- The feasibility of running advanced AI coding workflows on local hardware
- How architectural decisions, UI quality, and feature completeness differ across models
- The economics of cloud vs. self-hosted AI for autonomous development

The goal is not to pick a winner for marketing purposes, but to surface genuine, reproducible data about where these models excel and where they fall short.
